## 一、项目概述

本项目是一个基于 Telegram 的区块链交易机器人系统，采用 **Monorepo + 微服务** 架构，面向 DEX（去中心化交易所）链上交易场景。用户通过 Telegram Bot 交互，完成钱包导入、行情查询、下单交易及自动化策略管理。

**技术栈概览：**

| 层级 | 技术 | 用途 |
|------|------|------|
| 前端交互 | Node.js + Grammy + TypeScript | Telegram Bot 对话与菜单 |
| 核心引擎 | Go | 高并发交易执行、钱包签名、行情聚合 |
| 管理后台 | Next.js | 运营与风控管理 |
| 基础设施 | Docker、K8s、Redis、PostgreSQL | 部署与数据持久化 |

## 三、架构与数据流

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Telegram User  │────▶│   bot-service    │────▶│ wallet-service  │
│                 │     │   (Grammy/TS)    │     │  (Go, gRPC)     │
└─────────────────┘     └────────┬─────────┘     └─────────────────┘
                                 │
                    ┌────────────┼────────────┐
                    ▼            ▼            ▼
           ┌──────────────┐ ┌──────────────┐ ┌─────────────────┐
           │trading-engine│ │market-service│ │strategy-service │
           │ (Go, 订单)   │ │ (行情聚合)   │ │ (跟单/SLTP)     │
           └──────────────┘ └──────────────┘ └─────────────────┘
                    │            │            │
                    └────────────┼────────────┘
                                 ▼
                    ┌──────────────────────────┐
                    │  Redis / PostgreSQL      │
                    │  (缓存、Session、订单)   │
                    └──────────────────────────┘
```

- **bot-service**：用户唯一入口，解析命令与按钮回调，通过 gRPC/HTTP 调用其他服务。
- **wallet-service**：管理私钥加密存储、余额查询、交易签名。
- **trading-engine**：接收交易指令，与 DEX 链上交互，执行买卖。
- **market-service**：聚合行情，供 Bot 展示价格、计算滑点等。
- **strategy-service**：实现跟单、限价单、狙击、止盈止损等自动化策略。

## 五、开发流程与阶段划分

### 阶段 0：环境与 Monorepo 搭建

1. **完善 pnpm-workspace.yaml**  
   - 声明 `apps/*`、`packages/*` 工作区

2. **配置 turbo.json**  
   - 定义 `build`、`dev` 等 pipeline，启用缓存

3. **完善 .env.example**  
   - `BOT_TOKEN`、`DATABASE_URL`、`REDIS_URL`、`MASTER_KEY`（KMS 主密钥占位）

4. **根 package.json**  
   - 添加 `pnpm install`、`turbo run build` 等脚本

---

### 阶段 1：协议与数据层

1. **proto 定义**  
   - `wallet.proto`：GetBalance、RequestSign 等  
   - `trading.proto`：CreateOrder、CancelOrder、GetOrders 等  
   - `market.proto`：GetPrice、SubscribePrice 等  
   - `strategy.proto`：CreateCopyTrade、SetSLTP 等

2. **生成代码**  
   - `buf` 或 `protoc` 生成 Go / TS 存根到 `packages/generated/`

3. **数据库设计与迁移**  
   - 撰写 `docs/database-design.md`（E-R 图、表结构）  
   - 实现 `infra/migration/001_users.sql`、`002_wallets.sql`、`003_orders.sql`、`004_audit_logs.sql`

4. **bus-schema**  
   - 定义 Redis Pub/Sub 消息格式（如 `order.filled`、`price.update`）

---

### 阶段 2：核心后端服务（按依赖顺序）

1. **wallet-service**  
   - 账户派生、私钥加密存储（vault 抽象）  
   - gRPC：GetBalance、RequestSign  
   - 开发期可用 mock_vault，生产接入 KMS

2. **market-service**  
   - 多 DEX 价格聚合、缓存到 Redis  
   - gRPC：GetPrice、ListTokens  
   - 可选：Honeypot 检测模块

3. **trading-engine**  
   - 依赖 wallet（签名）、market（价格）  
   - 订单状态机、DEX 交互、Gas 策略、风控（滑点、黑名单）  
   - gRPC：CreateOrder、CancelOrder、GetOrderStatus

4. **strategy-service**  
   - 依赖 trading-engine  
   - 实现跟单、限价单、狙击、SLTP 等策略  
   - 监听 Redis 事件（如价格变动、成交通知）

5. **audit-service**  
   - 记录私钥解密、资金流向、风控告警  
   - 可异步消费 Redis 或直接写入 PostgreSQL

---

### 阶段 3：Bot 服务完善

1. **目录与中间件**  
   - `middlewares/`：鉴权、频率限制、日志  
   - `sessions/`：Redis Session 存储  
   - `types/`：Context 扩展与类型定义

2. **命令与菜单**  
   - `commands/`：`/start`、`/wallet`、`/buy`、`/sell`、`/orders`、`/help`  
   - `keyboards/`：Inline Keyboard 模板  
   - `menus/`：持仓列表、订单列表翻页

3. **多步对话**  
   - `conversations/`：导入私钥、设置止损等引导流程

4. **API 客户端**  
   - `api/grpc.client.ts`、`api/http.client.ts` 调用 wallet、trading、market

5. **事件监听**  
   - `listeners/`：订阅 Redis 成交通知，推送给用户

6. **工具与国际化**  
   - `utils/`：金额格式化、地址脱敏  
   - 可选：`packages/common-ts` 的 i18n 支持

---

### 阶段 4：基础设施与部署

1. **Docker**  
   - 各服务 Dockerfile  
   - `docker-compose.yml` 本地一键启动

2. **Webhook 模式**  
   - `nginx.conf` 反向代理 Telegram Webhook  
   - bot-service 支持 Webhook 模式运行

3. **K8s（可选）**  
   - 各服务 Deployment、Service、ConfigMap、Secret

---

### 阶段 5：管理后台与扩展

1. **frontend**  
   - Next.js 管理后台  
   - 用户列表、订单监控、风控告警、审计日志查询

2. **监控与告警**  
   - Prometheus + Grafana  
   - 关键指标：订单成功率、延迟、Gas 消耗

## 七、安全与风控要点

- **私钥**：仅以密文存储，使用 KMS 或 AES 加密，生产禁用 mock_vault  
- **签名**：采用即用即焚逻辑，签名完成后不持久化明文私钥  
- **风控**：滑点限制、黑名单地址、单笔/单日限额、夹子检测  
- **审计**：所有私钥解密、资金操作、风控告警写入 audit 表
